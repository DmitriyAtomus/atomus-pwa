# Синхронизация «Деловых Линий» через PAT

Фронтенд CRM использует только собственный авторизованный API. Он не должен
обращаться к `api.dellin.ru` напрямую: иначе PAT окажется доступен JavaScript-коду
страницы, а запросы будут зависеть от CORS.

Официальный метод авторизации:
<https://dev.dellin.ru/api/auth/login/#_header3>.

## Настройка

`POST /api/settings/dellin`

```json
{
  "appkey": "<ключ приложения Деловых Линий>",
  "pat": "<персональный токен с префиксом dl-api->"
}
```

Обработчик должен:

1. проверить авторизацию пользователя Atom CRM и право менять интеграции;
2. вызвать `POST https://api.dellin.ru/v4/auth/login.json` с `appkey` и `pat`;
3. убедиться, что в ответе получен `data.sessionID`;
4. сохранить `appkey` и PAT только в серверном хранилище, недоступном клиенту;
5. никогда не возвращать PAT, appkey или sessionID во фронтенд и логи;
6. вернуть `{ "ok": true, "message": "Деловые линии подключены" }`.

PAT действует 365 дней. При истечении сессии сервер повторно получает
`sessionID` по сохранённым `appkey + pat`.

## Чтение и обновление

`GET /api/logistics/dellin` возвращает только безопасное состояние:

```json
{
  "configured": true,
  "shipments": []
}
```

`POST /api/logistics/dellin/refresh`:

1. получает или обновляет `sessionID`;
2. запрашивает накладные через API «Деловых Линий»;
3. обновляет у локальных отправлений `auto_status`, `auto_from`, `auto_cargo`,
   ожидаемую дату и безопасный `sync_error`;
4. возвращает `{ "synced": 0, "total": 0 }`.

Ошибки API нужно очищать от значений с префиксом `dl-api-` до логирования и
возврата клиенту.

## Миграция со старой схемы

Поля `login` и `password` больше не принимает и не показывает фронтенд.
Сохранённые ранее парольные данные следует удалить после успешного сохранения
PAT. До этого сервер может продолжать читать старую конфигурацию только для
плавной миграции.
